# Observability & Monitoring -- Interview Q&A Guide

This document contains commonly asked interview questions related to
Monitoring, Observability, and Metrics in modern cloud-native
environments such as Kubernetes-based systems.

------------------------------------------------------------------------

# 1. What is the difference between Monitoring and Observability?

### Monitoring

Monitoring focuses on **detecting when something goes wrong** in a
system. It relies on predefined metrics, dashboards, and alerts.

Examples: - CPU usage threshold alerts - Memory utilization alerts - Pod
restart alerts in Kubernetes

Monitoring answers the question:

**"When did the system fail or behave abnormally?"**

### Observability

Observability goes deeper and helps understand **why a system is in its
current state**. It provides insights using telemetry data such as:

-   Metrics
-   Logs
-   Traces

Observability answers the question:

**"Why did the system fail and what caused the issue?"**

In short:

  Monitoring         Observability
  ------------------ -------------------------
  Detects problems   Explains the root cause
  Alert-based        Investigation-based
  Reactive           Analytical

# 2. How do you emit custom logs and metrics in an application?

### Logging

Applications typically emit logs at different levels:

-   INFO
-   DEBUG
-   WARN
-   ERROR

For example, in Java applications developers commonly use:

-   **Log4j**
-   **Logback**
-   **SLF4J**

These logs capture runtime behavior such as exceptions, debug
information, and operational messages.

### Custom Metrics

Custom metrics are created by instrumenting the application code.

Common approaches:

-   Prometheus client libraries
-   OpenTelemetry instrumentation

Example (Python using Prometheus client):

``` python
from prometheus_client import start_http_server, Counter

REQUEST_COUNT = Counter('app_requests_total', 'Total request count')

start_http_server(8000)

def process_request():
    REQUEST_COUNT.inc()
```

In many modern systems, developers use **OpenTelemetry** to generate
metrics, logs, and traces which can then be exported to observability
platforms.

# 3. What kind of metrics do you work with in your project?

In most Kubernetes-based production systems, metrics are collected at
multiple levels.

### Infrastructure Metrics (Node Exporter)

Examples:

-   node_cpu_seconds_total
-   node_memory_MemAvailable_bytes
-   node_disk_io_time_seconds_total
-   node_network_receive_bytes_total

These metrics help monitor the **health of nodes and infrastructure**.

### Kubernetes Metrics

Collected from kubelet, kube-state-metrics, or the Kubernetes API.

Examples:

-   container_cpu_usage_seconds_total
-   container_memory_usage_bytes
-   kube_pod_container_status_running
-   kube_node_status_condition

These metrics provide visibility into **cluster and container health**.

### Application Metrics

Custom metrics generated by the application itself.

Examples:

-   http_requests_total
-   http_request_duration_seconds
-   db_query_duration_seconds
-   payment_failures_total

These metrics help measure **business logic performance and
reliability**.

# 4. Have you worked on Observability? Explain your implementation.

Yes. Observability in our system was implemented using the three core
telemetry pillars:

-   Metrics
-   Logs
-   Traces

### Logs

We collected application logs using logging frameworks such as
**Log4j**.

Pipeline used:

Application → Logstash → Elasticsearch → Kibana

This setup (ELK Stack) helped analyze application behavior and debug
errors using searchable logs.

### Metrics

Metrics were collected using:

-   Prometheus
-   Node Exporter
-   Kubernetes exporters

Custom application metrics were added through **OpenTelemetry
instrumentation**.

Grafana dashboards were created to visualize:

-   Request volume
-   Service latency
-   Error rates
-   Payment request failures

### Tracing

Distributed tracing was implemented using **Jaeger**.

Tracing helped identify:

-   Service-to-service communication
-   Latency between microservices
-   Slow database queries

For example, when the payment service slowed down during high traffic,
tracing helped identify the bottleneck in a downstream service.

# 5. What is the difference between Metrics, Logs, and Traces?

### Logs

Logs capture **detailed events that happen inside an application at a
specific time**.

Examples:

-   error stack traces
-   debug messages
-   transaction events

Logs are best for **deep debugging**.

------------------------------------------------------------------------

### Metrics

Metrics are **numerical measurements collected over time**.

Examples:

-   request rate
-   CPU usage
-   memory utilization
-   response latency

Metrics are best for **monitoring trends and triggering alerts**.

------------------------------------------------------------------------

### Traces

Traces show the **complete journey of a request across multiple
services** in distributed systems.

Example flow:

User → API Gateway → Payment Service → Database

Tracing helps identify:

-   where latency occurs
-   which service failed
-   dependency relationships

# 6. What is the difference between Push and Pull based monitoring?

There are two main monitoring models.

### Pull-based Monitoring

Used by **Prometheus**.

In this model, the monitoring system **scrapes metrics from targets** at
regular intervals.

Example Prometheus configuration:

``` yaml
scrape_configs:
  - job_name: "myapp"
    static_configs:
      - targets: ["app1:8000", "app2:8000"]
```

Advantages:

-   Easier service discovery
-   Better reliability
-   Centralized control

------------------------------------------------------------------------

### Push-based Monitoring

In this model, applications **push metrics to a monitoring gateway or
collector**.

Examples:

-   StatsD
-   Telegraf

Example:

``` bash
echo "myapp.request:1|c" | nc -u -w1 statsd.local 8125
```

Advantages:

-   Useful for short-lived jobs
-   Works well in batch environments

# Summary

A complete observability stack usually includes:

  Component    Tool
  ------------ ------------------------
  Metrics      Prometheus
  Dashboards   Grafana
  Logs         ELK Stack
  Tracing      Jaeger / OpenTelemetry

Together these tools help teams monitor systems, investigate incidents,
and understand system behavior in production environments.
